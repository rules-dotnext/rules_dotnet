# Architecture

Technical overview of rules_dotnet internals for contributors.

## TFM Transitions (Central Complexity)

Target Framework Moniker (TFM) transitions are the mechanism that makes
multi-targeting work. Every rule with a `deps` attribute applies a Bazel
configuration transition on dependencies.

### How it works

1. A `csharp_library` declares `target_frameworks = ["net8.0", "net9.0"]`.
2. Bazel creates one configured target per TFM (split transition).
3. When another target depends on this library, the transition inspects
   the consumer's TFM and selects the best match from the dependency's
   `target_frameworks` list.
4. Selection algorithm: exact match > highest compatible TFM > fail.

The compatibility chain is defined in `dotnet/private/common.bzl` via
`FRAMEWORK_COMPATIBILITY`. It encodes the .NET TFM hierarchy:
`netstandard` < `netcoreapp` < `net5.0` < ... < `net10.0`.

### Configuration fan-out

A binary targeting `net9.0` that depends on a library with
`target_frameworks = ["net8.0", "netstandard2.0"]` gets the `net8.0`
variant. The transition fires at every edge in the dependency graph,
creating per-TFM sub-graphs automatically.

This is analogous to `--platforms` transitions in rules_go/rules_cc:
users never write `select()` for TFMs.

### Key files

- `dotnet/private/transitions/tfm_transition.bzl`: The transition implementation
- `dotnet/private/common.bzl`: `FRAMEWORK_COMPATIBILITY`, `get_highest_compatible_target_framework()`
- `dotnet/private/macros/register_tfms.bzl`: TFM config settings generated by `register_tfms()`, invoked from `dotnet/BUILD.bazel`

## The Publish Pipeline

Publishing transforms a `csharp_binary` into a deployable artifact.

```
csharp_binary ──> publish_binary ──> apphost + deps.json + runtimeconfig.json
                                 └─> (optional) self-contained with runtime pack
                                 └─> (optional) single-file bundle

csharp_library ──> publish_library ──> flat directory with deps.json (no apphost)

csharp_binary ──> native_aot_binary ──> standalone native executable via ILC
```

### publish_binary

1. Receives `DotnetBinaryInfo` from the upstream binary.
2. Generates `deps.json` from transitive `DotnetAssemblyRuntimeInfo` providers.
3. Generates `runtimeconfig.json` with framework version and roll-forward policy.
4. Creates an apphost launcher from `DotnetApphostPackInfo`.
5. For self-contained: bundles the runtime pack's DLLs and native files.
6. For single-file: runs the .NET single-file bundler.

### deps.json generation

`generate_depsjson()` in `common.bzl` walks the transitive runtime
dependency graph and emits the JSON structure the .NET host uses to
probe for assemblies. Key complexity: when self-contained, runtime pack
DLLs can override user-provided NuGet DLLs based on version comparison.

## NuGet Resolution Model

Two entry points, same underlying machinery:

### Paket path
```
paket.lock ──> paket2bazel ──> paket.main_extension.bzl ──> nuget_repo ──> import_library per package
```

### NuGet lock file path
```
packages.lock.json ──> nuget module extension ──> parse_nuget_lock_file() ──> nuget_repo ──> import_library
```

### nuget_archive (repository rule)

Downloads a `.nupkg`, extracts it, analyzes the NuGet package layout
(`lib/`, `ref/`, `runtimes/`, `analyzers/`, `tools/`, `contentFiles/`),
and generates a BUILD file with TFM-aware `select()` aliases.

### nuget_repo (repository rule)

Creates a Bazel repository from a list of package descriptors. For each
package, it generates an `import_library` target with the appropriate
`deps`, `refs`, `libs`, `native`, and `analyzers` attributes.

### import_library (rule)

The leaf target that provides `DotnetAssemblyCompileInfo` and
`DotnetAssemblyRuntimeInfo`. Handles targeting pack overrides,
framework list version comparisons, and meta-package auto-exports.

## Provider Flow

Providers are the data contracts between rules.

```
                    ┌─ DotnetAssemblyCompileInfo (refs, analyzers, exports)
csharp_library ─────┤
                    └─ DotnetAssemblyRuntimeInfo (libs, native, resource_assemblies)

                    ┌─ DotnetBinaryInfo (dll, transitive_runtime_deps, pack infos)
csharp_binary  ─────┤
                    └─ DotnetAssemblyRuntimeInfo

                    ┌─ DefaultInfo (published files)
publish_binary ─────┤
                    └─ RunfilesInfo

dotnet_analysis_config ──── DotnetAnalysisConfigInfo (analyzer_files, configs, warning policy)
```

### Strict deps

When `strict_deps = True` (default), compilation only sees direct deps'
`refs`. When `False`, `transitive_refs` are also included. The `exports`
mechanism allows a library to explicitly re-export a dependency's refs
without disabling strict deps globally.

### Compile info collection

`collect_compile_info()` in `common.bzl` is the central function that
resolves what assemblies a compilation target can see. It handles:
- Targeting pack overrides (system assemblies replace NuGet versions)
- Framework list version comparisons
- Export propagation
- Analyzer collection (common + language-specific)
- Content source injection from source-only NuGet packages

## Toolchain

The `dotnet` module extension downloads per-platform SDK archives and
registers toolchains. The toolchain provides:
- `csc` / `fsc` compilers
- `dotnet` runtime
- Runtime TFM and version info
- Strict deps build setting

SDK versions are pinned in MODULE.bazel via `dotnet.toolchain()`.
